# 解像度FM — 実装プラン

## Context

ユーザーは「世界の解像度を上げ、仮説思考・洞察力を高め、人生を楽しくしたい」。出来事の表面ではなく深層のダイナミズム(背景・構造)を読めるようになりたい。そのための思考トレーニングツールを新規開発する。

最重要要件は **「作ったのに使わなくなる」を絶対に防ぐこと**。スマホでスキマ時間にポチポチでき、ラジオ感覚で聴ける、楽しく一生使い続けられる形にする。

**引き継ぎ要件(ユーザー明示)**: Fable 5 の利用は限られているため、他モデルに切り替わっても品質が落ちないよう、リポジトリ内ドキュメント・メモリーファイルを徹底整備すること。

## 確定した設計決定(ユーザーとのQ&Aで合意済み)

| 論点 | 決定 |
|---|---|
| 中核体験 | ①出来事の「なぜ」推理 ②メンタルモデル1日1個 ③ラジオ解説 の3本柱を1つの日次番組に統合 |
| AI利用方式 | **事前生成型**。スマホ側にAPIキー不要・追加課金ほぼゼロ。仮説を書いた後に模範分析を開封して自己採点する方式 |
| ホスティング | **GitHub Pages(公開リポジトリ)** + Claude Codeスケジュール機能(クラウドエージェント)で毎日自動生成・自動コミット |
| 音声 | まず**ブラウザ内蔵TTS(Web Speech API)**。台本形式で作るので後から自然音声MP3に差し替え可能な構造にする |
| 継続の仕組み | ストリーク+記録 / メンタルモデル図鑑(収集) / 過去の自分との比較 / 毎日変わる題材の鮮度 / **魅力的なキャラクター(毎日話したくなる存在)** — 全部入り |
| キャラクター | 2人掛け合い形式。MCは**フェイ**(Fable由来、ユーザー指定)+後輩ヒナタ。名前・性格はユーザーが設定画面で変更可能に |
| 分析レンズ | 固定4レンズは「見方の固定化」リスクありとのユーザー指摘により、**レンズ・ライブラリ(初期~15本、拡張可)から毎日3〜5本を動的選択**する方式に変更。開封前にユーザーが「効きそうなレンズ」を予想する工程を追加 |
| メンタルモデル | 100個は初期カリキュラム。**エージェントがリストを拡張し続ける**(ユーザー指定)。全モデル配信後は深化モード(組み合わせ・反例・限界) |
| 設計承認 | 2026-07-02 に承認済み。「他のモデルが引き継いでも大丈夫なように」との追加指示あり |

## プロダクト概要

名称(仮): **解像度FM** — 毎朝AIが「今日の番組」を自動生成して届けるパーソナルラジオ局PWA。

### 毎日の体験(5〜10分)
1. **今日の推理**(約3分): 最近の実際の出来事を1つ提示 → ガイド質問(「誰が得をする?」「お金はどう流れる?」「構造か偶然か?」等)にタップで答え+自由記述で自分の仮説を構築 → **「この出来事、どのレンズが効きそう?」をレンズ・ライブラリから予想選択** → 開封すると**その日の出来事に合わせて選ばれた3〜5本のレンズによる多層分析**が読める(自分のレンズ予想との答え合わせつき) → 「自分に抜けていた視点」をタップして自己採点 → 仮説ログに蓄積
2. **今日のレンズ**(約2分): メンタルモデルを1日1個。約100個からスタートするマスターリスト(エージェントが継続的に拡張)から体系的に配信。図鑑にコレクション。今日の推理と絡めた適用ミニクイズつき。全モデル配信後は深化モード(モデルの組み合わせ・反例・限界を扱う2周目)へ移行
3. **ラジオモード**: 上記2つを2キャラの掛け合い台本(8〜10分)として聴ける。キャラごとに声(pitch/rate)を変えて読み上げ、再生中の行をハイライト。読む/聴くは同一コンテンツの別モード
4. **週次(土日)**: 今週のモデル復習クイズ+過去仮説の振り返り(クライアント側ロジックで実現、生成不要)

### キャラクターバイブル(GENERATION.mdに収録)
- **フェイ**(MC、Fable由来・ユーザー命名): 博識だが断定しない。軽妙。口癖「面白い見方ですね。でも逆から見ると…」。多層的視点の体現者
- **ヒナタ**(後輩): 好奇心旺盛。リスナーの代弁者。「えっ、なんでですか!?」と率直に聞く。素朴な疑問が深掘りの起点
- 台本にはストリーク・継続への言及ができる汎用セリフ枠(クライアントが差し込む変数 `{{streak}}` 等)を用意

## アーキテクチャ

```
[Claude Code クラウドエージェント(毎朝 JST 05:30 頃)]
   Web検索で題材選定 → content/YYYY-MM-DD.json 生成 → 検証 → commit/push
        ↓
[GitHub リポジトリ(公開) → GitHub Pages 自動配信]
        ↓
[スマホの PWA] fetch で当日JSONを取得。個人データは localStorage のみ(公開されない)
```

- ビルド工程なしの vanilla HTML/CSS/JS(ES modules)。AIが保守しやすく壊れにくい
- 生成失敗日のフォールバック: `content/evergreen/` に常備エピソード。当日ファイルが無ければそれを表示し、ストリークは途切れない扱い
- 個人データはlocalStorage + 設定画面からJSONエクスポート/インポート(機種変更・消失対策)

## 継続メカニクス仕様
- **ストリーク**: 1日1アクション(推理の自己採点完了 or ラジオ完聴 or クイズ回答)でカウント。**2日連続で空けたときのみ途切れる**猶予設計。累計思考回数も表示
- **図鑑**: models.json全体をシルエット表示、配信済みは開放。ミニクイズ正解や後日の復習正解で習得度が☆1→☆3に育つ
- **過去の自分**: ログに約30日前のエントリがあればホームに「1ヶ月前のあなたはこう考えていた」カードを表示
- **キャラ反応**: 台本内の変数枠+クライアント側でストリーク・図鑑進捗に応じたホームの一言セリフ

## 実装ステップ

1. **スキャフォールド**: `人生/kaizodo-fm` 作成、git init、`gh repo create`(public)、GitHub Pages有効化(mainブランチ root)
2. **コンテンツ基盤**: JSONスキーマ確定、`data/models.json`(初期約100モデル: 経済/心理/システム/歴史/確率統計/戦略のカテゴリバランス)、`data/lenses.json`(初期~15本)、`data/characters.json`、`scripts/validate.mjs`、**GENERATION.md**(品質の要。実例台本つきで厚めに書く。以下を明記: レンズは毎日3〜5本を動的選択し時々意外なレンズを混ぜる/モデルリストは残り30個を切ったらエージェントが拡張する/全モデル配信後は深化モード/題材ジャンルのローテーションと過去回重複回避)
3. **PWAアプリ実装**: frontend-designスキルを使いモバイルファーストで実装。深夜ラジオスタジオ的な落ち着いたダークトーン基調。タブ: ホーム/推理/レンズ/ラジオ/図鑑/記録。store.js→各画面の順
4. **初期コンテンツ生成**: 当日分1本+evergreen 2本をGENERATION.mdに従い生成、validate通過を確認
5. **PWA仕上げ・デプロイ**: manifest/sw.js、push、Pages URLで動作確認。ユーザーにURLを渡しスマホで「ホーム画面に追加」してもらう
6. **日次自動化**: /schedule スキルで毎朝JST 05:30頃のルーティン作成。プロンプトは「リポジトリのGENERATION.mdに従い当日エピソードを生成→validate→commit/push」のみ(手順の実体はリポジトリ側に置き、どのモデルでも同品質にする)。注: クラウドエージェント実行はユーザーのClaudeサブスクリプション使用量を消費する旨を伝える
7. **引き継ぎ整備**: リポジトリCLAUDE.md作成+**メモリーファイル作成**(下記)

## メモリーファイル(ステップ7で作成、プランモード解除後すぐ)
- `project: kaizodo-fm-project` — プロジェクトの目的・場所・アーキテクチャ・スケジュールルーティンの存在
- `user: user-profile` — 思考力向上が目標、日本語、楽しさ・継続性重視、スマホ利用
- `feedback: maintain-quality-across-models` — モデル切替時も品質維持のため、判断基準はリポジトリ内ドキュメント(GENERATION.md等)に外部化し、メモリーを常に最新化するという方針
- `MEMORY.md` インデックス更新

## 検証方法
- `node scripts/validate.mjs content/2026-07-02.json` がパスすること
- ローカルHTTPサーバ(`python -m http.server`等)でPC・ブラウザのモバイル表示にて全タブ動作確認: 仮説→開封→自己採点→ログ保存、クイズ、図鑑開放、ラジオ再生(読み上げ・ハイライト・キャラ別声)、ストリーク計算(localStorageを日付操作してテスト)、エクスポート/インポート
- push後、GitHub Pages URLで実機同等の確認。当日JSONが無い日付でのevergreenフォールバック動作確認
- スケジュールルーティンを1回手動実行(run now)し、翌日分が正しく生成・コミット・配信されることを確認
