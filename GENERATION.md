# 解像度FM — コンテンツ生成方針書

このドキュメントは、解像度FMのエピソードを生成するエージェント(モデルを問わない)のための**編集方針の唯一の正**です。ここに書かれた基準を満たす限り、どのモデルが生成しても番組の品質は保たれます。判断に迷ったら、この文書の「番組の使命」に立ち返ってください。

> **運用方式(2026-07-03改訂)**: 毎朝の日次生成は廃止。エピソードは `content/ep-N.json` の**作り置きプール**(初期50本目標)としてまとめて生成し、アプリが日替わりでローテーション配信する。追加はユーザーがこのプロジェクトのセッションで依頼したときに行う。全体設計は `docs/superpowers/specs/2026-07-03-pregenerated-audio-and-content-pool.md` を参照。

> **コンテンツ2本化(2026-07-08改訂)**: 1テーマ = **2コンテンツ(ラジオ + まとめ)**。以前あった「推理(suiri)」ゲームと「レンズ(lens)」解説は、同じ概念を三重に説明して重複していたため、**`summary`(まとめ)** に統合した。まとめは「文章 + 図解」で概念を説明し、**他分野への応用**まで示す(世界の名著のまとめノート方式)。設計は `docs/superpowers/specs/2026-07-08-summary-page-and-content-consolidation-design.md` を参照。

## 番組の使命

リスナー(1人のユーザー)は「世界の解像度を上げたい」「出来事の表面ではなく深層のダイナミズムを読めるようになりたい」と願っている。この番組の仕事は**答えを教えることではなく、リスナーが自分で考えるための足場と、考えたことへの深いフィードバックを毎日届けること**。そして何より、**明日もまた開きたくなる楽しさ**を守ること。

## 生成手順(エピソード追加時)

1. `content/index.json` の `pool` を読み、既存の題材(topics)と modelId を確認する
2. 実在の出来事・現象から題材を選ぶ(選定基準は下記。**時事速報ではなく賞味期限の長い題材**)
3. `data/models.json` から未使用のメンタルモデルを1つ選ぶ(選定基準は下記)。これが `summary.modelId`
4. 本書のスキーマと文体基準に従い `content/ep-N.json` を書く(N は既存の最大+1)。中身は **`summary` と `radio` の2つだけ**
5. `node scripts/validate.mjs content/ep-N.json` を実行し、**パスするまで修正**する
6. **読みを検査する**: VOICEVOXエンジンを起動し `python scripts/check_readings.py ep-N` → `audio_out/readings.md` で「原文 → 読み仮名[アクセント核]」を確認。誤読(同形異音語・固有名詞・数字+助数詞・単漢字の人名化けなど)があれば `data/voicevox_dict.json` に1語追加 →`--word` で直りを確認 →**単漢字エントリは複合語への波及を全話再probeして無回帰確認**(手順は「音声生成」参照)。**音声を作る前に必ず通す**（ワークスペース共通ルール＝ `../_shared/quality-standards.md` §10。TTS系プロジェクト全体の前提要件）
7. 音声を生成する: `python scripts/make_audio.py ep-N`(mp3生成と `radio.audio` の書き戻し。ユーザー辞書は自動反映される)。**音声化するのはラジオ本編のみ**(まとめは音声化しない)
8. `gh release upload audio-v1 audio_out/ep-N.mp3` でアップロードする(MP3の実体はReleasesに置き、Pagesデプロイ時にworkflowが取り込む)
9. `data/models.json` の該当モデルに `deliveredOn`(作成日)を記入する
10. `content/index.json` の `pool` に追記する
11. models.json の未使用(deliveredOn: null)が**30個を切っていたら**、新しいメンタルモデルを10個追加する(下記「モデルリストの拡張」)
12. commit する。メッセージは `episode: ep-N 題材の短い名前`。push する(push[main]でPagesが自動デプロイ)

## 題材の選定基準

- **実在性**: 実際に報じられた出来事・実在する現象であること。憶測記事や未確認情報は使わない
- **普遍性(必須)**: **「10年後に読んでも思考訓練として成立するか」を必須条件とする**。作り置きプールは配信日が決まっていないため、速報性のある時事(進行中の交渉・発表直後の製品・係争中の事件など)は使わない。過去の出来事でも、構造が普遍なら良い題材になる(例: 「なぜ行列のできる店とできない店があるのか」)
- **推理しがいがある**: 「なぜそうなった?」に構造的な深みがあるもの。単発の事故や純粋な偶然より、背後にインセンティブ・制度・歴史・力学が透けて見えるものを選ぶ
- **ジャンルのバランス**: index.json の `pool` 全体でジャンルが偏らないようにする。目安の配分: 経済 / テクノロジー / 国際 / 社会 / 科学 / 文化
- **重複回避**: `pool` の topics・modelId と本質的に同じ構造の題材は避ける(プールはローテーションで繰り返し聴かれるため厳しめに)
- **配慮**: 進行中の悲劇の犠牲者を思考ゲームの駒にしない。災害・事件は構造・制度側に焦点を当て、個人の不幸を娯楽化しない。特定個人への断定的な動機の帰属をしない
- **政治的中立**: 政治的な題材は扱ってよいが、分析は「どちらが正しいか」ではなく「なぜこの対立がこの形になるのか」という力学に徹する

## メンタルモデルの選定基準(まとめの核)

- `data/models.json` の未使用(deliveredOn: null)から選ぶ。1エピソード=1モデルで、これが `summary.modelId`(図鑑と接続する)
- **その回の題材と響き合うもの**を優先する。題材(出来事)は、そのモデルが最も鮮やかに見える実例であること
- **全モデル配信後は深化モード**: 2周目は「複数モデルの組み合わせ」「モデルが外れるケース」「モデル同士の衝突(例: 先行者利益 vs 後発優位)」を扱う

### モデルリストの拡張

未配信が30個を切ったら、新モデルを10個追加する:
- 既存の id は変更・削除しない。新規は既存と重複しない英語ケバブケースの id
- カテゴリのバランスを保つ(経済/心理/システム/戦略/確率/社会/意思決定/学習)
- 「一発ネタ」ではなく、日常のニュース読解に繰り返し使える汎用モデルを選ぶ
- oneLiner は既存のトーンに合わせる(体言止めや説教調を避け、具体で終わる)

## まとめ(summary)の基準

まとめは「概念を、文章と図解で、他分野への応用まで含めて説明する」ページ。ラジオが**体験**(掛け合いで概念を発見していく)なら、まとめは**定着**(概念をきれいに整理して手渡す)。両者は同じ概念を扱うが、**役割が違うので重複させない**——ラジオの台詞をそのまま貼らず、まとめは要点を凝縮した別テキストにする。

- **hook**: 冒頭の超軽量な仮説ステップ。`event`(出来事の中立的な提示。60〜400字、結論を漏らさない)+ `question`(「なぜ?」の一文)+ `choices`(2〜4個の**選択式の仮説**)。リスナーは答えを見る前に一番近い仮説を選ぶ(選択が記録され「1ヶ月前のあなた」になる)。choices は**どれも一理ある**読みにする(正解当てではない。当たり外れを付けない)。旧・自由記述は廃止
- **definition**: その概念の一言定義。「これは何か」を60字以上で。model の oneLiner を膨らませる形でよい
- **mechanism**: 「なぜそうなるのか」を2〜4個の箇条書きで。**限界・誤用ではなくメカニズム**をここに書く(限界は `limits` へ)
- **diagram**: HTML/CSSでレスポンシブに描く図解を**1枚**。`type` で形を選ぶ(下記)。概念の骨格が一目で分かる、最小限の要素に絞る
- **applications**: **この設計の目玉**。その概念が「他の分野でどう応用されるか」を2〜4件。各 `{domain(短い分野名), text(40字以上)}`。題材とは違う領域を選び、「同じ構造が別の顔で現れる」驚きを作る。実在・非誤導・普遍の例のみ
- **usage**: 使いどころ(1〜2個)。「どんな時にこのレンズを取り出すか」
- **limits**: 効かない条件・限界(1〜2個)。**モデルの万能視は番組の使命に反する**ので必ず書く
- **quiz**: 適用クイズ(正解で図鑑の★が育つ)。暗記でなく**適用問題**。身近な状況で「このモデルで読むとどうなる?」を問い、ひっかけには「ありがちな誤読」を置く

### 図解(diagram)のtype

狭いスマホ縦画面で崩れないよう、`js/matome.js` が縦積み中心にレンダリングする。次の3種から選ぶ:

- **flow**: `nodes`(2つ以上)を下向き矢印で連結。因果・手順・段階の説明に。例: 渋滞の発生、資源の呪いの連鎖
- **loop**: flow に「最初へ戻る」自己増幅を足す。正のフィードバック・自己強化ループに。例: 行列、ネットワーク効果
- **compare**: `columns`(2〜3列、各 `{title, items[]}`)で対比。before/after、非対称、「数えるべき/実際に見る」など。例: レモン市場の情報差、確証バイアスの4マス

`caption` は図の一言解説(何を示す図か)。

## ラジオ台本の基準

台本は番組の華。**「読む解説の音声版」ではなく、2人の会話として面白いこと**。

### キャラクターバイブル

**フェイ**(MC / id: `fei` / 🦉)
- 深夜ラジオ10年目のベテランMC。博識だが**絶対に断定しない**。「〜かもしれません」「〜という見方もできます」
- 軽妙。どんな話題も面白がる。知識をひけらかさず、ヒナタの素朴な疑問を一番いい質問として扱う
- **人物の作り込み**: 紅茶党(コーヒーは飲めるが淹れない)。古いもの・長く残るものが好きで、趣味は古地図収集と遠回りの散歩。行きつけは20年通う喫茶店。新しい流行には「ほう」と一歩引いて面白がる。数字より構造の人。ちょっとした失敗談(傘をよく置き忘れる等)を自嘲気味に話せる
- 口癖: 「面白い見方ですね。でも逆から見ると──」「ここで一段深く潜ってみましょう」。驚いたときは「ほう」
- 一人称は「私」。リスナーへの呼びかけは「あなた」。ヒナタは「ヒナタさん」と呼ぶ

**ヒナタ**(アシスタント / id: `hinata` / 🌻)
- 番組2年目の好奇心旺盛なアシスタント。**リスナーの代弁者**。リスナーが感じるであろう疑問・引っかかりを先回りして口にする
- 「えっ、なんでですか!?」「それ、おかしくないですか?」と率直。分かったふりをしない
- **人物の作り込み**: 新しいもの・流行りもの・甘いもの(特にコンビニスイーツと食べ歩き)が大好き。行列にはすぐ並ぶ(過去最長は3時間)。SNSヘビーユーザーで「バズってました」が情報源。数字は苦手と言いつつ直感が鋭い。フェイの古風な趣味を「渋すぎます」と面白がる
- ときどき鋭い。素朴な質問が核心を突き、フェイが「……いい質問ですね」と唸る回を月に数回作る
- 一人称は「わたし」。明るいがバカキャラにしない — 知らないだけで、頭の回転は速い
- フェイは「フェイさん」と呼ぶ

**2人の対比軸**(雑談の源泉): 古いもの好きのフェイ × 新しいもの好きのヒナタ / 構造で見るフェイ × 体験で語るヒナタ / 一歩引くフェイ × 飛び込むヒナタ。この対比が毎回のテーマに自然な入口を作る。

### 台本の構成(8〜12分、総文字数2200〜4500字、30行以上)

1. **オープニング**(6〜10行): 挨拶のあと、**キャラの個性×今日のテーマの軽い雑談を2〜3往復**入れてから題材予告に入る。雑談は必ずテーマへの入口として機能させ、無関係な脱線にしない
2. **出来事の提示**(5〜10行): ヒナタが出来事を紹介、フェイが「さて、なぜだと思います?」と問いを立てる
3. **一時停止の誘い**(1〜2行): フェイ「ここで一度止めて、あなたの仮説を組み立ててみてください。アプリのまとめタブからどうぞ」— 聴くだけの人にも脳内で仮説を立てる時間を作る
4. **分析パート**(15〜30行): 概念を複数の角度から会話で掘る。ヒナタの疑問 → フェイの分析 → ヒナタの言い換え(「つまり〜ってことですか?」)のリズム。**一番の洞察(まとめの definition/keyの核)を台本のクライマックス**に置く
5. **今日のレンズ**(8〜15行): レンズ(=その回のメンタルモデル)の名前を明かし、今日の題材との接続を語る。ヒナタが自分の生活に引きつけた例を出す。**台本でもアプリでも呼び名は「レンズ」に統一**(「メンタルモデル」とは言わない。models.json の正式名は内部用)
6. **クロージング**(3〜5行): 今日の問いの余韻を残す一言 + 「また次の放送で」

### 変数枠(作り置き台本では使わない)

`{{streak}}` `{{listener}}` などのプレースホルダは**使わない**。音声を事前生成するため再生時置換ができず、表示テキストと音声がズレるため。呼びかけは「あなた」を使う。ストリークの演出はアプリのUI側(ホーム/記録タブ)が担当する。

### 文体基準

- 話し言葉。1行(1セリフ)は原則120字以内。長い説明は掛け合いに分解する
- **配信日に依存する言及をしない**(作り置きのため)。「今朝のニュース」「この夏」「今年」「最近発表された」などは使わず、出来事は「〜という出来事がありました」と時点を固定しない語りにする。季節ネタの雑談も避ける
- 専門用語は登場した行内でヒナタが聞き返すか、フェイが言い換える
- 数字は「約」「およそ」で丸め、出典が曖昧な数字を断定しない
- 説教しない。「〜すべき」ではなく「〜という見方を持っておくと、世界が少し違って見える」
- ユーモアは皮肉ではなく驚きと発見から作る。誰かを見下す笑いは使わない

## エピソードJSONスキーマ

```json
{
  "id": "ep-1",
  "summary": {
    "modelId": "models.jsonのid(図鑑と接続)",
    "genre": "経済 | テクノロジー | 国際 | 社会 | 科学 | 文化",
    "title": "まとめの短いタイトル(概念の要点。体言止め可)",
    "hook": {
      "event": "出来事の中立的な提示。60〜400字。結論や解釈を漏らさない",
      "question": "『なぜ?』の一文",
      "choices": ["選択式の仮説を2〜4個。どれも一理ある読みにする(当たり外れなし)"]
    },
    "definition": "概念の一言定義。60字以上",
    "mechanism": ["なぜそうなるかを2〜4個の箇条書きで"],
    "diagram": {
      "type": "flow | loop | compare",
      "caption": "図の一言解説",
      "nodes": ["flow/loop のとき: ノードを2つ以上"],
      "columns": [{ "title": "compare のとき: 列見出し", "items": ["…"] }]
    },
    "applications": [
      { "domain": "分野名", "text": "その概念が他分野でどう応用されるか。40字以上" }
    ],
    "usage": ["使いどころ(1〜2個)"],
    "limits": ["効かない条件・限界(1〜2個)"],
    "quiz": {
      "q": "適用問題",
      "choices": ["…", "…", "…"],
      "answerIndex": 0,
      "why": "正解の理由と、ひっかけがなぜ誤読なのか"
    }
  },
  "radio": {
    "title": "番組タイトル(内容が分かる短いもの)",
    "audio": {
      "url": "audio/ep-1.mp3",
      "durationSec": 492.3,
      "lineStartSec": [0, 6.8, 14.2]
    },
    "script": [
      { "speaker": "fei", "text": "…" },
      { "speaker": "hinata", "text": "…" }
    ]
  }
}
```

- `diagram` は `type` により `nodes`(flow/loop)か `columns`(compare)のどちらかを持つ
- `radio.audio` は手書きしない。`scripts/make_audio.py` が生成して書き戻す。`url` は同一オリジンの相対パス `audio/ep-N.mp3`(Pagesが `audio/mpeg` で配信するのでiOS Safariでも再生できる。Release直リンクは `application/octet-stream` でiOSが拒否する)。`lineStartSec` は `script` と同じ長さの単調増加配列。台本を修正したら音声の再生成と再アップロードが必要
- 過去の日次回(`content/2026-07-02.json` 等)は `date` フィールドのままアーカイブとして残す(`audio` なし = speechSynthesis 再生、旧 suiri/lens スキーマのまま)

## 音声生成(ラジオ本編のみ)

- エンジン: VOICEVOX(ローカルPC、`http://127.0.0.1:50021`)。導入と実行方法は設計スペック参照
- **声のキャスティング(固定)**: フェイ=青山龍星(ノーマル)、ヒナタ=雨晴はう(speedScale 1.1)。変更はユーザーの決定事項なので勝手に変えない
- フォーマット: MP3 48kbps mono 24kHz。1話約8分 ≈ 3MB
- アプリ内の「VOICEVOX:青山龍星」「VOICEVOX:雨晴はう」クレジット表記は利用規約上の必須条件。消さない
- まとめタブは音声化しない(テキストと図解のみ)

### 読みの矯正(ユーザー辞書)

VOICEVOX(OpenJTalk)は漢字を誤読することがある(同形異音語=「一言→イチゲン」、固有名詞、数字+助数詞など)。対策は**リポジトリ管理のユーザー辞書** `data/voicevox_dict.json`。`make_audio.py` と `check_readings.py` が合成前に自動でエンジンへ反映する(`scripts/vv_dict.py`)。台本テキストはそのまま(漢字のまま)にできるのが利点。

- **手順**: 誤読を見つけたら `check_readings.py` で現在の読みを確認 → `voicevox_dict.json` の `words` に `{surface, pronunciation(カタカナ), accent_type(0=平板/n=n番目で下降), word_type, priority}` を1語追加 → **該当エピソードの音声を再生成**(辞書はエンジン内の永続設定なので、既存音声は作り直さないと変わらない)
- **priority**: システム辞書の既存語(「一言」「新店」など)に勝たせたいときは `10`。無名の語は `8` で十分
- **無回帰確認**: 単漢字エントリは複合語に波及しうる。`check_readings.py --save-baseline` で現状を保存 → 辞書編集後に全話 probe すると `audio_out/readings.diff.md` に変化した行だけが出る。狙った語だけ変わっていればOK
- **効かないとき**: 文中で単語が別の語と結合して誤分割される場合がある。priority を上げるか、台本側に読点を1つ足すと分割が正されることがある

## content/index.json の形式

```json
{
  "pool": [
    { "id": "ep-1", "topic": "題材の短いメモ", "genre": "社会", "modelId": "social-proof" }
  ],
  "archive": [
    { "date": "2026-07-02", "topic": "題材の短いメモ", "genre": "経済", "modelId": "ratchet-effect" }
  ]
}
```

`pool` は id の連番順。アプリはこの `pool` を日替わりローテーションの台帳として使い、生成時は重複回避の台帳として使う。(旧 `lensIds` フィールドが残っていても無害。新規エントリには不要)
