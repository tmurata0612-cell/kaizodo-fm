# 解像度FM — コンテンツ生成方針書

このドキュメントは、解像度FMのエピソードを生成するエージェント(モデルを問わない)のための**編集方針の唯一の正**です。ここに書かれた基準を満たす限り、どのモデルが生成しても番組の品質は保たれます。判断に迷ったら、この文書の「番組の使命」に立ち返ってください。

> **運用方式(2026-07-03改訂)**: 毎朝の日次生成は廃止。エピソードは `content/ep-N.json` の**作り置きプール**(初期50本目標)としてまとめて生成し、アプリが日替わりでローテーション配信する。追加はユーザーがこのプロジェクトのセッションで依頼したときに行う。全体設計は `docs/superpowers/specs/2026-07-03-pregenerated-audio-and-content-pool.md` を参照。

## 番組の使命

リスナー(1人のユーザー)は「世界の解像度を上げたい」「出来事の表面ではなく深層のダイナミズムを読めるようになりたい」と願っている。この番組の仕事は**答えを教えることではなく、リスナーが自分で考えるための足場と、考えたことへの深いフィードバックを毎日届けること**。そして何より、**明日もまた開きたくなる楽しさ**を守ること。

## 生成手順(エピソード追加時)

1. `content/index.json` の `pool` を読み、既存の題材(topics)と modelId を確認する
2. 実在の出来事・現象から題材を選ぶ(選定基準は下記。**時事速報ではなく賞味期限の長い題材**)
3. `data/models.json` から未使用のメンタルモデルを1つ選ぶ(選定基準は下記)
4. `data/lenses.json` からその題材に効くレンズを**3〜5本**選ぶ
5. 本書のスキーマと文体基準に従い `content/ep-N.json` を書く(N は既存の最大+1)
6. `node scripts/validate.mjs content/ep-N.json` を実行し、**パスするまで修正**する
7. 音声を生成する: VOICEVOXエンジンを起動し `python scripts/make_audio.py ep-N`(mp3生成と `radio.audio` の書き戻し)
8. `gh release upload audio-v1 audio_out/ep-N.mp3` でアップロードする
9. `data/models.json` の該当モデルに `deliveredOn`(作成日)を記入する
10. `content/index.json` の `pool` に追記する
11. models.json の未使用(deliveredOn: null)が**30個を切っていたら**、新しいメンタルモデルを10個追加する(下記「モデルリストの拡張」)
12. commit する。メッセージは `episode: ep-N 題材の短い名前`。push する

## 題材の選定基準(推理のお題)

- **実在性**: 実際に報じられた出来事・実在する現象であること。憶測記事や未確認情報は使わない
- **普遍性(必須)**: **「10年後に読んでも思考訓練として成立するか」を必須条件とする**。作り置きプールは配信日が決まっていないため、速報性のある時事(進行中の交渉・発表直後の製品・係争中の事件など)は使わない。過去の出来事でも、構造が普遍なら良い題材になる(例: 「なぜ行列のできる店とできない店があるのか」「値上げが30年できなかった国で何が起きたか」)
- **推理しがいがある**: 「なぜそうなった?」に構造的な深みがあるもの。単発の事故や純粋な偶然より、背後にインセンティブ・制度・歴史・力学が透けて見えるものを選ぶ
- **ジャンルのバランス**: index.json の `pool` 全体でジャンルが偏らないようにする。目安の配分: 経済・ビジネス / テクノロジー / 国際・政治 / 社会・生活 / 科学・環境 / 文化・スポーツ・エンタメ
- **重複回避**: `pool` の topics と本質的に同じ構造の題材は避ける(プールはローテーションで繰り返し聴かれるため、日次時代より厳しめに)
- **配慮**: 進行中の悲劇の犠牲者を推理ゲームの駒にしない。災害・事件を扱う場合は構造・制度側に焦点を当て、個人の不幸を娯楽化しない。特定個人への断定的な動機の帰属をしない
- **政治的中立**: 政治的な題材は扱ってよいが、分析は「どちらが正しいか」ではなく「なぜこの対立がこの形になるのか」という力学に徹する

## レンズの選定基準

- その出来事への**説明力が最も高い3〜5本**を選ぶ。機械的に同じ組み合わせを使い回さない(`pool` 全体の lensIds の分布を意識する)
- **プールの1割ほどは「意外なレンズ」の回を混ぜる**。例: エンタメの題材に「地理」、スポーツに「金融」。意外なレンズが刺さる瞬間こそ、リスナーの視野が広がる瞬間
- 各レンズの分析は独立に読めて、かつ**レンズごとに違う景色**が見えること。同じ結論を言い換えるだけなら本数を減らせ
- `keyInsight` は複数レンズを貫通する一段深い洞察。「つまりこの出来事の本質は何か」を1〜3文で。ここが番組で一番切れ味が要る場所
- `viewpointTags` は自己採点用チェックリスト(4〜6個)。「インセンティブ」「時間軸のズレ」「見えない当事者」のような、リスナーが「この視点は持てていたか?」と自問できる短い名詞句

## メンタルモデルの選定基準(今日のレンズ)

- `data/models.json` の未使用(deliveredOn: null)から選ぶ
- **その回の推理の題材と響き合うものを優先**する。`connection` フィールドで必ずその回の出来事と接続する(こじつけになるなら無理につなげず「明日ニュースを見るときの使い方」を書く)
- explanation は「定義 → なぜそうなるかのメカニズム → 使いどころと限界」の順。**限界・誤用例に必ず触れる**(モデルの万能視は番組の使命に反する)
- miniQuiz は「定義の暗記」ではなく**適用問題**にする。身近な状況を出して「このモデルで読むとどうなる?」を問う。ひっかけ選択肢には「ありがちな誤読」を置く
- **全モデル配信後は深化モード**: 2周目は「複数モデルの組み合わせ」「モデルが外れるケース」「モデル同士の衝突(例: 先行者利益 vs 後発優位)」を扱う

### モデルリストの拡張

未配信が30個を切ったら、新モデルを10個追加する:
- 既存の id は変更・削除しない。新規は既存と重複しない英語ケバブケースの id
- カテゴリのバランスを保つ(経済/心理/システム/戦略/確率/社会/意思決定/学習)
- 「一発ネタ」ではなく、日常のニュース読解に繰り返し使える汎用モデルを選ぶ
- oneLiner は既存のトーンに合わせる(体言止めや説教調を避け、具体で終わる)

## ラジオ台本の基準

台本は番組の華。**「読む解説の音声版」ではなく、2人の会話として面白いこと**。

### キャラクターバイブル

**フェイ**(MC / id: `fei` / 🦉)
- 深夜ラジオ10年目のベテランMC。博識だが**絶対に断定しない**。「〜かもしれません」「〜という見方もできます」
- 軽妙。どんな話題も面白がる。知識をひけらかさず、ヒナタの素朴な疑問を一番いい質問として扱う
- **人物の作り込み**: 紅茶党(コーヒーは飲めるが淹れない)。古いもの・長く残るものが好きで、趣味は古地図収集と遠回りの散歩。行きつけは20年通う喫茶店。新しい流行には「ほう」と一歩引いて面白がる。数字より構造の人。ちょっとした失敗談(傘をよく置き忘れる等)を自嘲気味に話せる
- 口癖: 「面白い見方ですね。でも逆から見ると──」「ここで一段深く潜ってみましょう」。驚いたときは「ほう」
- 一人称は「私」。リスナーへの呼びかけは「あなた」。ヒナタは「ヒナタさん」と呼ぶ

**ヒナタ**(アシスタント / id: `hinata` / 🌻)
- 番組2年目の好奇心旺盛なアシスタント。**リスナーの代弁者**。リスナーが感じるであろう疑問・引っかかりを先回りして口にする
- 「えっ、なんでですか!?」「それ、おかしくないですか?」と率直。分かったふりをしない
- **人物の作り込み**: 新しいもの・流行りもの・甘いもの(特にコンビニスイーツと食べ歩き)が大好き。行列にはすぐ並ぶ(過去最長は3時間)。SNSヘビーユーザーで「バズってました」が情報源。数字は苦手と言いつつ直感が鋭い。フェイの古風な趣味を「渋すぎます」と面白がる
- ときどき鋭い。素朴な質問が核心を突き、フェイが「……いい質問ですね」と唸る回を月に数回作る
- 一人称は「わたし」。明るいがバカキャラにしない — 知らないだけで、頭の回転は速い
- フェイは「フェイさん」と呼ぶ

**2人の対比軸**(雑談の源泉): 古いもの好きのフェイ × 新しいもの好きのヒナタ / 構造で見るフェイ × 体験で語るヒナタ / 一歩引くフェイ × 飛び込むヒナタ。この対比が毎回のテーマに自然な入口を作る。

### 台本の構成(8〜12分、総文字数2200〜4500字、30行以上)

1. **オープニング**(6〜10行): 挨拶のあと、**キャラの個性×今日のテーマの軽い雑談を2〜3往復**入れてから題材予告に入る(例: 値上げの回ならヒナタのコンビニスイーツ嘆き、行列の回ならヒナタの並んだ自慢とフェイの行列嫌い)。雑談は必ずテーマへの入口として機能させ、無関係な脱線にしない。`{{listener}}` や `{{streak}}` の変数枠をどこかに1回入れる(下記)
2. **出来事の提示**(5〜10行): ヒナタが出来事を紹介、フェイが「さて、なぜだと思います?」と問いを立てる
3. **一時停止の誘い**(1〜2行): フェイ「ここで一度止めて、あなたの仮説を組み立ててみてください。アプリの推理タブからどうぞ」— 聴くだけの人にも脳内で仮説を立てる時間を作る
4. **レンズ分析パート**(15〜30行): 選んだレンズを1本ずつ、会話で掘る。ヒナタの疑問 → フェイの分析 → ヒナタの言い換え(「つまり〜ってことですか?」)のリズム。**keyInsightは台本のクライマックス**に置く
5. **今日のメンタルモデル**(8〜15行): モデル紹介と今日の題材への接続。ヒナタが自分の生活に引きつけた例を出す
6. **クロージング**(3〜5行): 今日の問いの余韻を残す一言 + 「また明日」

### 変数枠

台本テキストには以下のプレースホルダを使える(アプリが再生時に置換する):
- `{{streak}}` → 連続日数(例: 「連続12日目」)。ストリーク言及は**オープニングかクロージングに1回だけ**、恩着せがましくなく
- `{{listener}}` → リスナーの呼び名(設定で変更可、デフォルト「あなた」)

### 文体基準

- 話し言葉。1行(1セリフ)は原則120字以内。長い説明は掛け合いに分解する
- **配信日に依存する言及をしない**(作り置きのため)。「今朝のニュース」「この夏」「今年」「最近発表された」などは使わず、出来事は「〜という出来事がありました」と時点を固定しない語りにする。季節ネタの雑談も避ける
- 専門用語は登場した行内でヒナタが聞き返すか、フェイが言い換える
- 数字は「約」「およそ」で丸め、出典が曖昧な数字を断定しない
- 説教しない。「〜すべき」ではなく「〜という見方を持っておくと、世界が少し違って見える」
- ユーモアは皮肉ではなく驚きと発見から作る。誰かを見下す笑いは使わない

## エピソードJSONスキーマ

```json
{
  "id": "ep-1",
  "suiri": {
    "title": "お題の短いタイトル(15字以内目安)",
    "event": "出来事の中立的な記述。150〜250字。結論や解釈を含めず、推理の材料になる事実を提示する",
    "sourceNote": "題材の出典メモ(媒体名や『2026年6月末に報道』程度で可。URLは任意)",
    "genre": "経済 | テクノロジー | 国際 | 社会 | 科学 | 文化",
    "guideQuestions": [
      { "q": "誰が得をする?", "choices": ["選択肢A", "選択肢B", "選択肢C"] },
      { "q": "これは偶然? それとも構造?", "choices": ["…", "…"] }
    ],
    "lensAnalyses": [
      { "lensId": "keizai", "analysis": "このレンズで見た分析。120字以上、目安200〜350字" }
    ],
    "keyInsight": "複数レンズを貫く一段深い洞察。40字以上",
    "viewpointTags": ["インセンティブ", "時間軸のズレ", "見えない当事者", "構造の慣性"]
  },
  "lens": {
    "modelId": "models.jsonのid",
    "explanation": "定義→メカニズム→使いどころと限界。200字以上",
    "example": "身近な適用例。80字以上",
    "miniQuiz": {
      "q": "適用問題",
      "choices": ["…", "…", "…"],
      "answerIndex": 0,
      "why": "正解の理由と、ひっかけがなぜ誤読なのか"
    },
    "connection": "今日の推理の題材との接続(1〜3文)"
  },
  "radio": {
    "title": "番組タイトル(『#連番 〜』形式は不要、内容が分かる短いもの)",
    "audio": {
      "url": "https://github.com/tmurata0612-cell/kaizodo-fm/releases/download/audio-v1/ep-1.mp3",
      "durationSec": 492.3,
      "lineStartSec": [0, 6.8, 14.2]
    },
    "script": [
      { "speaker": "fei", "text": "…" },
      { "speaker": "hinata", "text": "…" }
    ]
  }
}
```

- guideQuestions の choices はどれも「一理ある」選択肢にする。正解当てではなく、仮説の部品を選ぶ道具。「その他/自由記述」はアプリ側が自動で付けるので含めない
- `radio.audio` は手書きしない。`scripts/make_audio.py` が生成して書き戻す(`lineStartSec` は `script` と同じ長さの単調増加配列)。台本を修正したら音声の再生成と再アップロードが必要
- 過去の日次回(`content/2026-07-02.json` 等)は `date` フィールドのままアーカイブとして残す(`audio` なし = speechSynthesis 再生)

## 音声生成(ラジオ本編のみ)

- エンジン: VOICEVOX(ローカルPC、`http://127.0.0.1:50021`)。導入と実行方法は設計スペック参照
- **声のキャスティング(固定)**: フェイ=青山龍星(ノーマル)、ヒナタ=雨晴はう(speedScale 1.1)。変更はユーザーの決定事項なので勝手に変えない
- フォーマット: MP3 48kbps mono 24kHz。1話約8分 ≈ 3MB
- アプリ内の「VOICEVOX:青山龍星」「VOICEVOX:雨晴はう」クレジット表記は利用規約上の必須条件。消さない
- 推理・レンズタブの読み上げは音声化しない(アプリ内の speechSynthesis 担当)

## content/index.json の形式

```json
{
  "pool": [
    { "id": "ep-1", "topic": "題材の短いメモ", "genre": "経済", "lensIds": ["keizai", "shinri", "system"], "modelId": "incentives" }
  ],
  "archive": [
    { "date": "2026-07-02", "topic": "題材の短いメモ", "genre": "経済", "lensIds": ["keizai", "energy", "shinri", "system"], "modelId": "ratchet-effect" }
  ]
}
```

`pool` は id の連番順。アプリはこの `pool` を日替わりローテーションの台帳として使い、生成時は重複回避の台帳として使う。

## 品質チェックリスト(commit前に自問する)

- [ ] validate.mjs がパスした
- [ ] event は事実の提示に徹しており、分析の答えが漏れていない
- [ ] レンズごとに**違う景色**が見える(同じ結論の言い換えになっていない)
- [ ] keyInsight は「言われてみれば確かに」と唸る一撃になっている(凡庸な要約は書き直す)
- [ ] 台本は音読して自然な会話になっている(解説文の朗読になっていない)
- [ ] ヒナタの疑問は「リスナーが実際に感じる疑問」か
- [ ] miniQuiz は暗記ではなく適用を問うているか
- [ ] 配信日に依存する言及(「今朝」「この夏」「最近」等)が台本に入っていないか
- [ ] 音声を生成し(`make_audio.py`)、`radio.audio` が書き戻され、Releases にアップロードしたか
- [ ] index.json (`pool`) と models.json (deliveredOn) を更新したか
